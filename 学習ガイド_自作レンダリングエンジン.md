# 自作レンダリングエンジン 学習ガイド

このプロジェクトは **K2EngineLow だけ** を使って動いています。
モデル描画・スプライト描画・ライティングといった「便利な部分」は、あえて外してあります。
それらを **自分の手で作っていく** のがこの授業のゴールです。

---

## 0. まず今の状態を理解する

ビルドして実行すると、**灰色の画面** が出ます。これがスタート地点です。

### いま何が起きているのか（`main.cpp`）

```cpp
InitGame(...);              // ウィンドウと K2EngineLow を初期化
NewGO<Game>(0, "game");     // Game クラスを1つ作る

while (DispatchWindowMessage())   // ゲームループ
{
    g_engine->BeginFrame();     // フレーム開始（画面を灰色でクリア）
    g_engine->ExecuteUpdate();  // 全ゲームオブジェクトの Update を呼ぶ
    g_engine->ExecuteRender();  // 全ゲームオブジェクトの Render を呼ぶ
    g_engine->EndFrame();       // フレーム終了（画面に表示）
}
```

- `Game` クラス（`Game.h` / `Game.cpp`）は `IGameObject` を継承しています。
- `Start()` → 最初に1回、`Update()` → 毎フレーム、`Render()` → 毎フレーム呼ばれます。
- 今は `Render()` が空なので、灰色のクリア色しか見えません。**ここに絵を描いていきます。**

### 使える「材料」（K2EngineLow が持っているもの）

| 材料 | クラス | 何をするもの |
|---|---|---|
| モデル | `Model` | tkm ファイルを読み込んで描画する |
| スプライト | `Sprite` | 2Dの画像を描画する |
| カメラ | `g_camera3D` / `g_camera2D` | 視点・射影 |
| シェーダ | `Shader` | hlsl(.fx) を読み込む |
| 定数バッファ | `ConstantBuffer` | シェーダに値を渡す（ライトの色など） |
| 構造化バッファ | `StructuredBuffer` | シェーダに配列を渡す（ライト複数など） |
| 描画ターゲット | `RenderTarget` | 一旦テクスチャに描く（ディファードで使う） |
| フォント | `Font` | 文字を出す |

> **これらを組み合わせて `ModelRender` や `SpriteRender`、`SceneLight` を自作します。**
> （これらの「完成版」は先生が持っています。この配布プロジェクトには入っていません。自分で作るのがこの授業です。）

## 進め方のルール（大事）

1. **1ステップ = 1つの新しいこと**。欲張らない。
2. **毎回ビルドして画面で確認**する。動いてから次へ。
3. コードは **①まず動かす → ②1行ずつ意味を理解 → ③1箇所だけ改造** の順。
4. うまくいったら、その状態を残しておく（フォルダごとコピーで「セーブ」）。
5. どうしても詰まったら、まず自分で調べる → それでもダメなら先生に聞く。**答えを写すのではなく、理解する** こと。

## Phase 1 : フォワードレンダリング（マスト）

「1つ描くたびに、その場でライティング計算までやってしまう」一番素直な方式です。まずこれを完成させます。

### Step 1-1 : スプライトを1枚出す（2D）

- **やること**：`Sprite` を使って画面に画像を1枚出す。
- **使う材料**：`Sprite`, `sprite.fx`（`Assets/shader/` にある）
- **見える成果**：画面に絵が出る（＝自分のコードで初めてピクセルが出た！）

#### そもそも Sprite（スプライト）の役割って何？

`Sprite` は「**2Dの四角形（板）にテクスチャを貼って画面に描く**」ためのクラスです。中身はたった頂点4つの板ポリゴンですが、ゲームでは大活躍します。

| 使い道 | 例 |
|---|---|
| UI / HUD | 体力ゲージ、スコア、ボタン、ミニマップ |
| 画面演出 | タイトルロゴ、フェードイン/アウト（黒画像の透明度を変える） |
| フルスクリーンパス | 画面全体を覆う1枚のスプライトに「加工用シェーダ」を掛ける |

つまりスプライトは「2D画像を出す部品」であると同時に、**「画面全体にシェーダを掛けるための道具」** でもあります。この使い方は Phase 2 のディファードレンダリングやポストエフェクトで主役になるので、Phase 1 のうちにスプライトの仕組みをしっかり理解しておくと、後がとても楽になります。

#### 最小コード

```cpp
// Game.h
Sprite m_sprite;

// Game::Start()
SpriteInitData initData;
initData.m_ddsFilePath[0] = "Assets/modelData/utc_all2.DDS"; // 表示したい DDS 画像
initData.m_fxFilePath = "Assets/shader/sprite.fx";
initData.m_width  = 256;   // 表示サイズ（ピクセル）。0のままだと何も出ない！
initData.m_height = 256;
m_sprite.Init(initData);

// Game::Update()
m_sprite.Update(
    { 0.0f, 0.0f, 0.0f },      // 座標（画面中央が原点）
    Quaternion::Identity,      // 回転
    { 1.0f, 1.0f, 1.0f });     // 拡大率

// Game::Render(rc)
m_sprite.Draw(rc);
```

- **座標系**：画面の **中央が原点 (0, 0)**。X は右がプラス、Y は上がプラス。
- **pivot（基準点）**：`Update` の第4引数。`(0.5, 0.5)` で画像の中心が基準（デフォルト）。`(0.0, 0.0)` で左下基準。ゲージを左端から伸ばしたいときなどに使う。
- テクスチャは **DDS 形式のみ** 読めます（`m_ddsFilePath`）。

#### 半透明も出せる（アルファブレンディング）

`SpriteInitData::m_alphaBlendMode` を設定すると、後ろの絵と「混ぜて」描けます。

| モード | 計算 | 見た目・用途 |
|---|---|---|
| `AlphaBlendMode_None` | 上書き | デフォルト。後ろの絵は消える |
| `AlphaBlendMode_Trans` | `新しい色 × α + 後ろの色 × (1-α)` | **普通の半透明**。すりガラス、フェード、UIの背景 |
| `AlphaBlendMode_Add` | `新しい色 + 後ろの色` | **加算合成**。光、炎、魔法エフェクト（描くほど明るくなる） |

```cpp
initData.m_alphaBlendMode = AlphaBlendMode_Trans; // Init 前に設定
```

透明度の指定方法は2つ：
1. **テクスチャのアルファチャンネル**：画像自体が持つ透明度（文字やアイコンの縁など）。
2. **乗算カラー**：`m_sprite.SetMulColor({1, 1, 1, 0.5f});` ← 第4成分(w)が全体の透明度。**毎フレーム値を変えればフェードイン/アウトになる。**

> **のぞいてみよう**：`k2EngineLow/graphics/Sprite.cpp` の `InitPipelineState()`。
> `m_alphaBlendMode` に応じて `SrcBlend`（新しい色に掛ける係数）と `DestBlend`（後ろの色に掛ける係数）を切り替えているだけです。半透明の正体は「この2つの係数の設定」だと分かります。

### Step 1-2 : `SpriteRender` を自作する
- **やること**：座標・回転・拡大をメンバ変数として持ち、`Init / Update / Draw` を整理した自作クラス `SpriteRender` を作り、Step 1-1 のコードを包む。
- **見える成果**：`SetPosition()` `SetScale()` などを呼ぶだけで絵が動く・回る・大きさが変わる。
- **狙い**：「使いやすい部品にまとめる」というエンジンの考え方を体験する（詳しくは Step 1-4 の「委譲」を参照）。

### Step 1-3 : モデルを1体出す（3D）

- **やること**：`Model` で `Assets/modelData/unityChan.tkm` を読み込んで描画。
- **使う材料**：`Model`, `ModelInitData`, `model.fx`, `g_camera3D`
- **見える成果**：3Dキャラが画面に出る（カメラは `system.cpp` で初期設定済み）。

#### 最小コード

```cpp
// Game.h
Model m_model;

// Game::Start()
ModelInitData initData;
initData.m_tkmFilePath = "Assets/modelData/unityChan.tkm";
initData.m_fxFilePath  = "Assets/shader/model.fx";
m_model.Init(initData);

// Game::Render(rc)
m_model.Draw(rc);   // カメラを渡さなければ g_camera3D が使われる
```

- **注意**：`ModelInitData.m_computedAnimationVertexBuffer` は **使いません**（既定の nullptr のまま）。スキニングは頂点シェーダ側でやります。

#### `Model` の中身はどうなっている？

`Model` は「入れ物」で、実際のデータと描画処理は `MeshParts` が持っています。

```
Model                                （k2EngineLow/graphics/Model.h）
 ├─ m_worldMatrix   … ワールド行列（どこに・どの向きで・どの大きさで置くか）
 ├─ m_tkmFile       … tkm ファイル（頂点・マテリアル情報の元データ）
 ├─ m_skeleton      … ボーン（アニメーション用。今は気にしなくてよい）
 └─ m_meshParts     … ★本体。描画に必要な GPU リソース一式
      │              （k2EngineLow/graphics/MeshParts.h）
      ├─ SMesh の配列（メッシュ＝形状のかたまり。1体のモデルに複数入っていることがある）
      │    ├─ m_vertexBuffer      … 頂点バッファ（座標・法線・UV・接線…の配列）
      │    ├─ m_indexBufferArray  … インデックスバッファ（三角形の組み立て方）※マテリアルごとに1つ
      │    └─ m_materials         … マテリアル（使うシェーダ＋テクスチャのセット）
      ├─ m_commonConstantBuffer   … 共通定数バッファ（ワールド/ビュー/プロジェクション行列）→ b0
      ├─ m_expandConstantBuffer   … 拡張定数バッファ（★君がライト情報を入れる場所）→ b1
      └─ m_descriptorHeap         … 上のリソースたちを「シェーダのどの受け口に繋ぐか」の対応表
```

**なぜマテリアルごとにインデックスバッファが分かれているのか？**
1体のモデルでも「肌のテクスチャ」「服のテクスチャ」など材質が複数あるからです。GPU は「テクスチャを切り替えたら描き直し」なので、材質ごとに三角形をまとめて、材質の数だけドローコール（描画命令）を出します。

#### `Draw()` を呼ぶと実際に何が起きる？

`Model::Draw` → `MeshParts::Draw`（`k2EngineLow/graphics/MeshParts.cpp` の `Draw` 関数）と進みます。**この関数は必ず一度読んでください。** 描画の1フレームの正体がここに全部書いてあります。流れは：

1. ワールド/ビュー/プロジェクション行列を `m_commonConstantBuffer` に **CopyToVRAM**（CPU→GPU へ転送）
2. 拡張定数バッファが登録されていれば、それも CopyToVRAM
3. メッシュごと・マテリアルごとにループ：
   - マテリアルの `BeginRender` （使うシェーダ・パイプラインステートを GPU にセット）
   - ディスクリプタヒープをセット（テクスチャや定数バッファをシェーダの受け口に接続）
   - 頂点バッファ・インデックスバッファをセット
   - `DrawIndexedInstance` ＝ **ドローコール発行**（GPU が三角形を描き始める）

### コラム : CPU からシェーダへ値を渡す仕組み（ConstantBuffer と register）

Step 1-5 以降のライティングでは「ライトの色や向き」を C++ からシェーダに渡す必要があります。ここを理解すると Phase 4 まで一生使えるので、じっくり読んでください。

#### register の `b` って何？

シェーダ側には、CPU からデータを受け取るための「**受け口（レジスタ）**」が種類別・番号付きで用意されています。

| 記号 | 意味 | このプロジェクトでの例（model.fx） |
|---|---|---|
| `b0, b1, …` | **B**uffer : 定数バッファ（cbuffer） | `b0` = 行列, `b1` = 拡張定数バッファ（★君が使う） |
| `t0, t1, …` | **T**exture : テクスチャなどの読み取り資源 | `t0` = アルベド, `t1` = 法線マップ, `t2` = スペキュラマップ, `t3` = ボーン行列 |
| `s0, s1, …` | **S**ampler : テクスチャの読み方の設定 | `s0` = サンプラ |
| `u0, u1, …` | **U**nordered : 読み書き両用資源 | Phase 3 のコンピュートシェーダで登場 |

`cbuffer XXX : register(b1)` と書いたら「**b1 番の受け口に届いたデータをこの変数たちとして読む**」という意味です。CPU 側が b1 に何かを届けて、初めて値が入ります。

#### 既にある定数バッファ：`m_commonConstantBuffer`（b0）

「モデルがなぜ最初から正しい位置に描けているのか」の答えがこれです。

- **シェーダ側**：`Assets/shader/ModelVSCommon.h` の先頭にある
  ```hlsl
  cbuffer ModelCb : register(b0)
  {
      float4x4 mWorld;   // ワールド行列
      float4x4 mView;    // ビュー行列（カメラ）
      float4x4 mProj;    // プロジェクション行列（射影）
  };
  ```
- **CPU 側**：`k2EngineLow/graphics/MeshParts.h` の `SConstantBuffer` 構造体（メンバが完全に対応している事を確認！）。
  `MeshParts::Draw` が毎フレーム `CopyToVRAM` で GPU に送っています。
- **接続**：`MeshParts::CreateDescriptorHeaps` が `RegistConstantBuffer(0, m_commonConstantBuffer)` で「0 番＝b0」に登録しています。

つまり流れは **「C++ の構造体に値を書く → CopyToVRAM → register(bN) の cbuffer に届く」**。これだけです。

#### 自分の値を渡すには：拡張定数バッファ（b1）を使う

`ModelInitData` に「自分の構造体のアドレスとサイズ」を渡すだけで、エンジンが **b1 に毎フレーム自動転送** してくれます。

```cpp
// ① 渡したいデータを構造体にする（Game.h などに）
struct DirectionLight
{
    Vector3 direction;   // ライトの向き
    float   pad0;        // ※後述のパディング
    Vector3 color;       // ライトの色
    float   pad1;
};
DirectionLight m_light;

// ② Init 時にアドレスとサイズを登録
initData.m_expandConstantBuffer     = &m_light;
initData.m_expandConstantBufferSize = sizeof(m_light);
m_model.Init(initData);

// ③ 以後は毎フレーム、メンバを書き換えるだけで自動で GPU に届く
//    （Draw のたびに m_light の「今の値」がコピーされる）
m_light.direction = { 1.0f, -1.0f, 1.0f };
m_light.direction.Normalize();
m_light.color = { 1.0f, 1.0f, 1.0f };
```

シェーダ側（`model.fx`）には対応する cbuffer を **b1** で書きます：

```hlsl
cbuffer DirectionLightCb : register(b1)
{
    float3 ligDirection;   // C++ の direction に対応
    float3 ligColor;       // C++ の color に対応
};
```

> **⚠ 16バイト・アライメントの罠（ここで一度は全員ハマる）**
> 定数バッファは 16 バイト（float 4個）単位で区切られます。HLSL の `float3` の直後に次の `float3` を書くと、自動的に 4 バイトの隙間が入ります。**C++ 側の構造体にも同じ位置に `float pad;` を入れて、メモリ配置を一致させてください。** ずれると「ライトの色に変な値が入る」「方向がおかしい」という怪奇現象になります。迷ったら全部 `Vector4` / `float4` にするのが安全です。

`Sprite` もまったく同じ仕組みです（`SpriteInitData::m_expandConstantBuffer` → b1）。`sprite.fx` の `cbuffer cb : register(b0)` には mvp 行列と乗算カラーが届いています（CPU 側は `Sprite::Draw` を見よ）。

#### 迷子になったらこの3ファイル

| 見る場所 | 分かること |
|---|---|
| `Assets/shader/ModelVSCommon.h` | シェーダ側の受け口（b0 の cbuffer、t3 のボーン行列） |
| `k2EngineLow/graphics/MeshParts.cpp` の `Draw` | CPU→GPU 転送のタイミング（毎フレーム CopyToVRAM） |
| `k2EngineLow/graphics/MeshParts.cpp` の `CreateDescriptorHeaps` | どのリソースが何番の受け口に繋がるか |

### Step 1-4 : `ModelRender` を自作する

- **やること**：座標・回転・拡大 → ワールド行列を計算して `Model` を動かす自作クラス。
- **見える成果**：キャラを好きな位置に置ける・動かせる。

```cpp
// 使う側（Game.cpp）はこう書けるようにするのがゴール
m_modelRender.Init("Assets/modelData/unityChan.tkm");
m_modelRender.SetPosition({ 100.0f, 0.0f, 0.0f });
m_modelRender.Update();       // 中で Model::UpdateWorldMatrix を呼ぶ
m_modelRender.Draw(rc);       // 中で Model::Draw を呼ぶ
```

#### なぜわざわざクラスを作るのか？ ―「委譲」という考え方

`ModelRender` は自分では描画しません。**中に `Model` を持ち、仕事を丸投げ（＝委譲、delegation）します。** 一見無駄に見えますが、これには大きな意味があります。

1. **使う側のコードが短くなる**：`ModelInitData` の細かい設定を `ModelRender::Init` の中に隠せる。ゲームを作る人は tkm のパスを渡すだけでいい。
2. **変更に強くなる**：Phase 2 でディファードレンダリングに切り替えるとき、変更するのは `ModelRender` の中身だけ。**`Game.cpp` 側の「Init して SetPosition して Draw」は1行も変わりません。** エンジンの中身を差し替えても、ゲームのコードが壊れない――これがエンジンを作る意味そのものです。
3. **機能の追いだしができる**：「全モデル共通でやりたいこと」（ライト情報を毎フレーム渡す、影の設定をする…）を1か所に集められる。

> 本家 k2Engine の `ModelRender` もまさにこの作りです。「継承で機能を足す」のではなく「部品を中に持って使う」──**継承より委譲**は、レンダリングに限らず良い設計の基本形です。

### Step 1-5 : ライティング① アンビエント（環境光）

- **やること**：`model.fx` にアンビエントカラー用の cbuffer（b1）を追加し、ピクセルシェーダでテクスチャの色に掛ける。
- **使う材料**：上のコラムの「拡張定数バッファ」
- **見える成果**：モデル全体の明るさ・色味が C++ から変えられる。

#### 計算式と考え方

「どこから来たか分からない、そこら中で反射しまくった光」をひとことで「一律この明るさ」と近似したものがアンビエントです。

```hlsl
float4 albedoColor = albedoTexture.Sample(Sampler, In.uv);
float3 lig = ambientLight;              // 環境光（例 : float3(0.3, 0.3, 0.3)）
albedoColor.xyz *= lig;                 // 素材の色 × 光
return albedoColor;
```

#### どういう色を設定すると、どう見える？

| ambientLight の値 | 見た目 |
|---|---|
| `(1.0, 1.0, 1.0)` | テクスチャそのまま（ライティング無しと同じ） |
| `(0.3, 0.3, 0.3)` | 全体が一様に薄暗い。**陰影はまだ無い** |
| `(0.2, 0.25, 0.4)` | 青みがかる → 夜・月明かりの雰囲気 |
| `(0.4, 0.3, 0.25)` | 暖色に寄る → 夕方・室内の白熱灯 |
| `(0.05, 0.05, 0.05)` | ほぼ闇 → ホラーの土台（この後のライトが際立つ） |

**アンビエントは「シーンの底の明るさ」** です。この後のディフューズ・スペキュラは「アンビエントに足す」形になるので、ここを低くするほど光と影のコントラストが強いドラマチックな画になり、高くするほどフラットで明るいアニメ調に寄ります。

### Step 1-6 : ライティング② ディフューズ（拡散反射）

- **やること**：ディレクションライト（向きと色）を b1 に渡し、法線とライトの内積（ランバート）で明るさを計算。
- **使う材料**：拡張定数バッファ（ライトの向き・色）、`SPSIn` の `normal`（頂点シェーダが既にワールド空間の法線を渡してくれている）
- **見える成果**：光の当たる面が明るく、裏が暗くなる。**初めて「立体」に見える瞬間。**

#### 計算式

「面が光に正面を向いているほど明るい」を内積で表します（ランバート反射）。

```hlsl
float3 normal = normalize(In.normal);              // 補間で長さが縮むので必ず正規化
float t = max(0.0f, dot(normal, -ligDirection));   // 正面=1、真横=0、裏=マイナス→0に切る
float3 diffuse = ligColor * t;

float3 lig = ambientLight + diffuse;               // 環境光に足す
albedoColor.xyz *= lig;
```

- `-ligDirection` の意味：`ligDirection` は「光が進む向き」。内積には「面からライトへ向かう向き」が要るので符号を反転します。
- `max(0, …)` を忘れると、光の裏側がマイナス（真っ黒以下）になり、アンビエントまで食われます。

#### どう設定すると、どう見える？

| 設定 | 見た目 |
|---|---|
| `direction = (0, -1, 0)`（真上から） | 頭・肩だけ明るい。昼の屋外 |
| `direction = (1, -0.5, 1)`（斜め上から） | 顔に陰影が出て一番「それっぽい」定番。まずこれ |
| `direction = (0, 1, 0)`（真下から） | 顔が下から照らされる。ホラーの定番ライティング |
| `color = (1, 1, 1)` | 白色光。標準 |
| `color = (1.0, 0.6, 0.35)` | 夕日。橙色の面と暗い面のコントラストが出る |
| `color = (0.3, 0.35, 0.55)` | 月光。青白い夜 |

### Step 1-7 : ライティング③ スペキュラ（鏡面反射）

- **やること**：視線ベクトルと反射ベクトルで「テカリ」を計算。カメラ位置（`g_camera3D->GetPosition()`）を定数バッファに追加で渡す。
- **見える成果**：ハイライト（光沢）が出る。**材質感（ツルツル/マット）を初めて出せる。**
- **キーワード**：Phong / Blinn-Phong、視線ベクトル、ハーフベクトル、鏡面反射の強さ(power)。

#### 計算式（Phong）

「ライトが表面で鏡のように反射し、その反射方向が目に向いているほど強く光る」です。ここで Step 1-3 で用意されていた `In.worldPos` が活躍します。

```hlsl
float3 refVec = reflect(ligDirection, normal);         // 反射ベクトル
float3 toEye  = normalize(eyePos - In.worldPos);       // ピクセル→カメラの向き
float t = pow(max(0.0f, dot(refVec, toEye)), specPow); // 反射が目に向くほど1に近い
float3 specular = ligColor * t * specIntensity;

float3 lig = ambientLight + diffuse + specular;        // 3つ揃った！
```

#### power（specPow）で材質が決まる

| specPow | ハイライトの見た目 | 材質のイメージ |
|---|---|---|
| 2〜10 | 広くぼんやり | 布、つや消しプラスチック、肌 |
| 20〜50 | ほどよい光沢 | 塗装、革、プラスチック |
| 100〜 | 小さく鋭い点 | 金属、ガラス、濡れた表面 |

`pow` は「内積の結果を何乗するか」。乗数を上げるほど「ほぼ正面のときだけ光る」＝ハイライトが引き締まります。**diffuse は「素材の色」に掛かるが、specular は光そのものの色で足す**（白いハイライト）という違いも観察してください。

#### ここまでの3つをどう配合する？ ― ゲームの絵作り早見表

| 作りたいゲーム | ambient | diffuse(ライト色/向き) | specular | ねらい |
|---|---|---|---|---|
| 明るい屋外アクション | 0.4 前後・空色寄り | 白〜暖色、斜め上から | 控えめ (pow 20) | 健康的で見やすい標準の絵 |
| ホラー | 0.05 以下 | 弱い青白、低い角度から | 強め (pow 50+) | 闇の中で濡れた光だけが目立つ |
| アニメ調キャラゲー | 0.6+ と高め | 白、正面寄り（影を減らす） | ほぼ 0〜ごく弱く | フラットで可愛い。※ディフューズを段階化するとトゥーンになる（発展） |
| 夕暮れのアドベンチャー | 0.25・紫寄り | 橙 (1.0, 0.55, 0.3)、水平に近い角度 | 中 (pow 30) | 長い影と暖色のドラマ |
| メカ・レース | 0.3 | 白、上から | 強い (pow 100+) | 金属のギラつきが主役 |

数字はあくまで出発点です。**1つだけ変えて→ビルドして→見る** を繰り返して、目を鍛えてください。

### Step 1-8 : スペキュラマップ
- **やること**：テカリの強さをテクスチャで場所ごとに変える。`Assets/modelData/utc_spec.DDS` がスペキュラマップで、エンジンが **t2** に登録済み（`MeshParts::CreateDescriptorHeaps` 参照）。`model.fx` に `Texture2D<float4> specularMap : register(t2);` を追加してサンプリングし、Step 1-7 の specular に掛けるだけ。
- **見える成果**：金属部分だけテカる、布はテカらない、など**1体の中で質感の差**が出る。

### Step 1-9 : ノーマルマップ（法線マップ）
- **やること**：テクスチャに書かれた法線で、ポリゴンより細かい凹凸を表現。法線マップはエンジンが **t1** に登録済み。
- **使う材料**：`Assets/modelData/utc_nomal.DDS`（法線マップ）、`SPSIn` の `tangent` / `biNormal`（接空間の基底。頂点シェーダが用意済み）
- **見える成果**：モデルの表面に細かいディテール（凹凸の陰影）が出る。
- **キーワード**：接空間(tangent space)、TBN行列、接線(tangent)・従法線(binormal)。
- **考え方**：法線マップの色 (0〜1) を `× 2 - 1` で (-1〜+1) のベクトルに戻し、`tangent / biNormal / normal` を使ってワールド空間に変換 → それを Step 1-6, 1-7 の `normal` の代わりに使う。
- **ここが Phase 1 の山場**。TBN の考え方を丁寧に。

> **Phase 1 のゴール**：`ModelRender` / `SpriteRender` / `SceneLight`（ライト情報をまとめて管理するクラス）が自作できていて、
> アンビエント＋ディフューズ＋スペキュラ＋スペキュラマップ＋ノーマルマップで、それっぽくキャラが描けている状態。
> `SceneLight` のヒント：ライトの構造体を1つにまとめ、全 `ModelRender` の拡張定数バッファ(b1)に同じものを渡せば、シーン共通のライトになります。

## Phase 2 : ディファードレンダリング

ライトの数が増えるとフォワードは重くなります。そこで「**まず素材だけ描く → あとでまとめて光を計算**」に変えます。

- **考え方**：
  1. **ジオメトリパス**：モデルの「アルベド（色）」「法線」「深度」「スペキュラ」などを、それぞれ別の `RenderTarget` に描く（これを G-Buffer と呼ぶ）。
  2. **ライティングパス**：画面いっぱいの四角形を1枚描き、G-Buffer を読みながらライティングを計算する。
- **使う材料**：`RenderTarget`（複数枚 = MRT）、フルスクリーン用の `Sprite`/四角形
- **見える成果**：見た目は Phase 1 と近いが、**ライトを何十個も置けても軽い**。
- **キーワード**：G-Buffer、MRT(複数レンダーターゲット)、深度から座標の復元、フルスクリーンパス。

---

## Phase 3 : TBR（タイルベースレンダリング）

ディファードをさらに速くする方法です。**ここで初めてコンピュートシェーダを使います。**

- **考え方**：画面を小さなタイル（例：16×16ピクセル）に分割し、
  「そのタイルに影響するライトだけ」を事前に絞り込んで(カリング)からライティングする。
- **使う材料**：コンピュートシェーダ、`RWStructuredBuffer`（タイルごとのライトリスト）
- **見える成果**：大量のライトがあっても、さらに軽くなる。
- **キーワード**：タイル分割、ライトカリング、コンピュートシェーダ、スレッドグループ。
- **難易度高**。Phase 2 が完全に理解できてから。

---

## Phase 4 : PBR（物理ベースレンダリング）

光の反射を「物理的に正しく」近似して、リアルな質感を出します。

- **考え方**：これまでの「なんとなくのスペキュラ」を、物理モデル（Cook-Torrance / GGX）に置き換える。
- **パラメータ**：メタリック（金属度）、ラフネス（表面の粗さ）、アルベド。
- **発展**：IBL（画像ベースライティング。スカイボックスから環境の映り込みを計算）。
- **見える成果**：金属・プラスチック・肌などが「本物っぽく」見える。
- **キーワード**：BRDF、Cook-Torrance、GGX、フレネル、メタリック/ラフネス、IBL。

## 発展課題（チャレンジ）

Phase 1〜4 ができた人へ。ここから先は「余裕がある人・もっと極めたい人」向けの追加テーマです。
**答えはこのプロジェクトには入っていません**（Phase 1〜4 と同じく、自分で作るのが課題です）。

| テーマ | 内容 | どのPhaseの後にやると良いか |
|---|---|---|
| **シャドウマップ** | ライト視点で深度を描き、影を落とす。まずハードシャドウ、次にソフトシャドウ | Phase 1 or 2 の後 |
| **スカイボックス** | 空（キューブマップ）を背景として描画する | Phase 1 の後 |
| **IBL（画像ベースライティング）** | スカイボックスを光源として使い、環境の映り込みを表現 | Phase 4 の後 |
| **ポストエフェクト：ブルーム** | 明るい部分をにじませて光をまぶしく見せる（ガウシアンブラーを使う） | Phase 2 の後 |
| **ポストエフェクト：トーンマップ** | HDR（広い明るさ）を画面用に圧縮する | ブルームと一緒に |
| **ポストエフェクト：被写界深度(DoF)** | ピントの外をぼかす | 上記の後 |
| **ポストエフェクト：SSAO** | 物のすき間や接地部を暗くして立体感を出す | Phase 2 の後 |
| **ポストエフェクト：SSR** | 画面情報から映り込み（反射）を作る | Phase 2 or 4 の後 |

> ヒント：ポストエフェクトはどれも「一度 `RenderTarget` に描いた画面を、もう一度シェーダで加工して描き直す」という共通の流れです。
> Phase 2 でやる `RenderTarget` の使い方が土台になります。

## 困ったときは

- **画面が真っ暗／何も出ない**：カメラの位置、モデルのスケール、シェーダの読み込みパスを疑う。
- **シェーダを直したのに変わらない**：ビルドし直したか？ `.fx` の保存を忘れていないか？
- **ビルドが通らない**：エラーメッセージの **一番上** を読む。だいたい原因はそこ。
- **どうしても分からない**：先生に相談する。答えを丸写しせず、**なぜそう書くのか** を理解してから自分のコードに反映する。

頑張ってください。**「灰色の画面」から始めて、最後は PBR まで自分で作れるようになる** のがこの授業です。
